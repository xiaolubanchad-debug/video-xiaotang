# video-xiaotang 开发计划

## 1. 规划目标

基于现有的 `VIDEO_CMS_PLAN.md`，先落地一个可上线的 MVP 版本：支持前台展示、视频详情播放、基础用户体系、后台 CMS 内容管理与推荐位配置。后续再扩展会员、广告、转码、统计报表等能力。

## 2. 关键假设

- 第一版以“点播型视频网站”做设计，不做短视频刷流模式
- 第一版优先支持后台录入视频地址，不把复杂转码作为首发阻塞项
- 前后台使用同一套代码仓库，降低启动成本
- 后台只保留一个超级管理员账号，不做多管理员角色系统
- `openclaw` 通过内部 API 实时新增、更新、删除视频内容
- 优先交付 CMS 可运营能力，而不是一次性做完商业化模块

## 3. 技术路线

- 前端框架：Next.js App Router
- 后端接口：Next.js Route Handlers
- 数据库：PostgreSQL
- ORM：Prisma
- 鉴权：NextAuth
- 样式：Tailwind CSS
- 管理后台：自建 Admin 页面
- 文件存储：本地开发先走本地/对象存储抽象，线上接 OSS/S3
- 视频播放：前期直接接入现成播放器，后续再扩展多清晰度与字幕
- 采集接入：内部受保护 API + API Key 鉴权 + 幂等写入

## 5. 里程碑开发计划

### 里程碑 0：项目初始化

目标：把工程底座搭起来，确保后面可以持续开发。

交付内容：

- 初始化 Next.js 项目
- 接入 Prisma 和 PostgreSQL
- 建立基础目录结构
- 建立环境变量模板
- 建立 UI 基础样式和设计令牌
- 建立 ESLint / 格式化 / 基础提交规范

完成标准：

- 项目可本地启动
- 数据库可连接
- 有首页占位和后台入口占位

### 里程碑 1：账户与权限

目标：打通用户身份体系和后台访问控制。

交付内容：

- 管理后台登录页

- 普通用户无法访问后台
- 管理员可以登录后台
- 会话状态稳定

### 里程碑 2：CMS 核心内容模型

目标：先把“内容可被管理”这件事做完整。

交付内容：

- `videos`
- `categories`
- `tags`
- `video_tags`
- `banners`
- `ingest_logs`

- 后台可完整新增和编辑视频
- 前台能读取视频数据并展示
- 分类、标签能参与筛选

### 里程碑 3：前台基础站点

## 6. 推荐开发顺序

1. 先搭工程和数据库
2. 再做超级管理员登录
3. 然后做内容模型、采集 API 和 CMS 视频管理
4. 接着做前台首页、分类页、详情页
5. 再补推荐位、评论审核、系统设置和采集日志
6. 最后补用户中心、收藏、历史和媒体增强

## 9. 风险与控制

- 风险：一开始就做转码、支付、广告会把项目拖慢
- 控制：这些放到第二阶段，首发只做可运营的内容站
- 风险：后台模块过多导致页面风格不统一
- 控制：先做一套共享 Admin 组件库
- 风险：视频播放来源复杂
- 控制：第一版统一走标准视频源模型，避免播放器层过早复杂化
- 风险：采集数据重复或脏数据污染内容库
- 控制：对 `source_provider + source_external_id` 做唯一约束，并保留采集日志和错误重试策略
